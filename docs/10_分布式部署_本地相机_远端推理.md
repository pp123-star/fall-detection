# 10 ‒ 分布式部署:本地相机 + 远端推理

> 客户端跑摄像头 + YOLO Pose,通过 WebSocket 把骨架上传到 autodl 服务器,服务器跑 PoseConv3D + 全套报警策略,把结果实时回传给客户端显示。

---

## 一、为什么这样设计?

### 1.1 直接传视频不可行的原因

朴素方案 (客户端传视频 → 服务器全推理 → 服务器传带 overlay 视频回来) 有两个致命问题:

| 路径 | 带宽 | 实际情况 |
|---|---|---|
| 客户端 ↑ 1080p H.264 视频 | 4-8 Mbps | 家庭宽带上行 10-50 Mbps,有抖动 |
| 服务器 ↑ 带 overlay 视频 | 4-8 Mbps | **autodl 等 GPU 租赁服务器到客户端的下行带宽往往 < 1 Mbps,这是瓶颈** |

服务器上行慢,所以反向传视频不通。

### 1.2 关键设计反转:数据流细切

把"重活留服务器,轻活留客户端"反过来思考 — **不传画面,只传"模型真正需要的最小信息":骨架点 (51 个浮点数 / 人)**。

```
方案 B 带宽对比 (30 FPS, 5 人):
  上行骨架 JSON:  ~ 0.28 Mbps  (300 倍小于视频)
  下行结果 JSON:  ~ 0.01 Mbps  (服务器上传瓶颈彻底绕开)
```

视频本身**只在客户端显示**,不进网络。这是整个设计的核心。

## 二、架构

```
┌─── 本地客户端 (笔记本/PC) ────────┐         ┌─── 远端服务器 (autodl 4090) ──────┐
│                                    │         │                                    │
│  cv2.VideoCapture (摄像头)         │         │  websockets.serve (8765)          │
│           │                        │         │           │                        │
│           ▼                        │         │           ▼                        │
│  YOLO Pose (CPU/GPU)               │         │  Per-Session FallDetectionSession │
│  yolo11s-pose.pt (~20MB)           │         │  内部 = MultiTrackFallDetector    │
│           │                        │         │  + PoseConv3D                      │
│           ▼  骨架 + bbox + tid     │         │           │                        │
│  build_frame_msg() ────────────────┼──ws──▶ │  process_frame()                   │
│                                    │ 0.3Mbps │     │                              │
│  asyncio recv_loop ◀───────────────┼──ws─── │     │ ┌─→ Result JSON              │
│           │                        │ 0.01M  │     │ │  (track 状态)              │
│           ▼                        │         │     │ └─→ Alert JSON (异步推送)   │
│  draw_overlay() (本地画 bbox/骨架) │         │                                    │
│           │                        │         │                                    │
│           ▼                        │         │                                    │
│  cv2.imshow() (本地显示)           │         │                                    │
│                                    │         │                                    │
└────────────────────────────────────┘         └────────────────────────────────────┘
```

### 2.1 客户端职责
- 摄像头采集
- 本地 YOLO Pose (提取骨架 + bbox + track_id)
- 通过 WebSocket 把骨架上传 (~0.3 Mbps)
- 接收服务器返回的状态 + 报警事件
- 在本地视频帧上画 overlay (bbox / 骨架 / FALL 标签 / 顶部红条)

### 2.2 服务端职责
- 接受 WebSocket 连接,每连接一个 session
- 每 session 独立维护 `MultiTrackFallDetector` (track 状态隔离)
- 收到一帧骨架 → 喂 `detector.update()` → 回 result JSON
- 报警事件通过 `SessionEventLogger` 异步推送

### 2.3 复用现有代码
**服务端 100% 复用** `inference.multitarget_realtime_demo.MultiTrackFallDetector` + 所有策略 (FallTrendDetector / PoseHeuristicScorer / TrackMerger / AlertPolicy / lost_track 等)。换句话说,**所有救 fall_7 的策略在分布式下都生效**。

## 四、部署步骤

### 4.1 服务端 (autodl)

```bash
# 1. 进入项目根目录,安装额外依赖
cd /root/autodl-tmp/fall-detection
pip install -r deploy/requirements_server.txt
# 现有 fall-detection 环境已经有 PyTorch + MMAction2,不需要重装

# 2. 启动服务端 (默认端口 8765,所有保护策略全开)
python -m deploy.server \
    --host 0.0.0.0 --port 8765 \
    --config configs/posec3d_fall_binary.py \
    --ckpt work_dirs/posec3d_fall_binary/best_acc_top1_epoch_5.pth \
    --device cuda:0

# 看到日志 "Listening on ws://0.0.0.0:8765" 就 OK
```

**autodl 端口暴露** (关键):

autodl 实例默认不能直接暴露 TCP 端口。两种办法:

**方法 A: SSH 反向隧道 (推荐,简单稳定)**

在你本地电脑上跑:
```bash
# 建立隧道:把本地 8765 转发到远端 8765
ssh -L 8765:localhost:8765 -CN \
    -p <autodl-ssh-port> root@<autodl-ssh-host>

# 客户端就连 ws://localhost:8765 即可
```

**方法 B: 用 autodl 的"自定义服务"功能**

autodl 控制台找到实例的"自定义服务"或"端口转发",把 8765 暴露为公网端口,然后:
```bash
python -m deploy.client --server ws://<autodl-public-host>:<public-port>
```

### 4.2 客户端 (本地电脑)

```bash
# 1. clone 项目 (或者只拷 deploy/ 目录)
cd ~
git clone <your-repo-or-just-copy-deploy-folder>

# 2. 创建一个轻量 Python 环境
python3 -m venv falldet_client
source falldet_client/bin/activate
pip install -r fall-detection/deploy/requirements_client.txt

# 3. 下载 YOLO Pose 权重
#    CPU 用户用 nano/small,GPU 用户随意
wget https://github.com/ultralytics/assets/releases/download/v8.3.0/yolo11s-pose.pt

# 4. 启动客户端
python -m deploy.client \
    --server ws://localhost:8765 \
    --pose-weights yolo11s-pose.pt \
    --camera 0 \
    --device cpu \
    --imgsz 640

# 摄像头窗口会打开,顶部状态条 + 实时 bbox/骨架/概率显示
# 摔倒时:顶部红条 "FALL DETECTED!" + 底部 reason 文字
```

## 六、性能预期

### 6.1 客户端 YOLO 性能 (CPU/GPU)

| 模型 | 大小 | 笔记本 i5/i7 CPU | M1/M2 Mac | RTX 30/40 系列 |
|---|---|---|---|---|
| yolo11n-pose.pt | 6 MB | 25-35 FPS | 30-40 FPS | 80+ FPS |
| yolo11s-pose.pt | 22 MB | 12-18 FPS | 18-25 FPS | 60+ FPS |
| yolo11m-pose.pt | 50 MB | 5-8 FPS | 10-15 FPS | 40+ FPS |
| yolo26x-pose.pt | 270 MB | 不可用 | ~5 FPS | 30+ FPS |

**推荐**: 笔记本/无 GPU 选 `yolo11n` 或 `yolo11s`,精度差距不大但 CPU 上能跑 15-30 FPS。

### 6.2 网络带宽 (实测预期)

| 链路 | 30 FPS × 5 人 | 备注 |
|---|---|---|
| 客户端 → 服务端 (frame msg) | ~ 0.3 Mbps | 单人 ~ 1.2 KB/帧 |
| 服务端 → 客户端 (result + alert) | ~ 0.05 Mbps | 服务器上传瓶颈完全无压力 |

即使带宽差到 1 Mbps 的环境也能稳定工作。

### 6.3 端到端延迟

| 段 | 典型耗时 |
|---|---|
| 客户端 YOLO 推理 | 30-80 ms (CPU yolo11s) |
| 上行网络传输 | 5-30 ms (autodl 国内节点) |
| 服务端 PoseConv3D 推理 | 10-20 ms (RTX 4090) |
| 下行网络传输 | 5-30 ms |
| 客户端渲染 | < 5 ms |
| **总计** | **60-170 ms** (符合实时要求) |

报警最差延迟 < 0.2 秒,人体跌倒地面后大约 1 秒内仍可触发,完全够用。

## 七、关键设计点

### 7.1 异步事件分离

客户端两个独立循环:
- **采集 + 发送循环**: 摄像头 → YOLO → 上行
- **接收循环**: `recv_loop()` 在后台异步消费结果和报警

两个循环用 `server_state` (dict) + `recent_alerts` (deque) 共享数据。这样**渲染始终用最新结果**,即使网络抖动也不卡顿。

### 7.2 服务端模型共享

所有 session **共享一个全局 `_GLOBAL_PREDICTOR`**:
```python
_GLOBAL_PREDICTOR = CachedClipPredictor(action_model, device=device)
```

这是节省显存的关键,否则每个客户端都加载一份 PoseConv3D 会爆显存。但每个 session **独立** `MultiTrackFallDetector` 实例 (track 状态、buffer、报警计数都不能共享)。

### 7.3 阻塞调用放线程池

`detector.update()` 内部调 PoseConv3D 是阻塞的 (PyTorch 同步),放在事件循环里会卡死其它 session。服务端关键代码:

```python
# server.py 主循环
loop = asyncio.get_event_loop()
result = await loop.run_in_executor(None, session.process_frame, msg)
await websocket.send(json.dumps(result))
```

`run_in_executor` 把同步调用扔到默认线程池,事件循环继续接其它客户端的消息。

### 7.4 报警事件的同步→异步桥接

`MultiTrackFallDetector` 的 `event_logger.log()` 是同步接口 (从 update() 内调用),但 WebSocket 是异步。解决:

```python
class SessionEventLogger:
    def log(self, ...):
        self.session.pending_alerts.append({...})   # list.append 线程安全

# 主循环每次发完 result 后:
alerts = session.drain_pending_alerts()
for alert in alerts:
    await websocket.send(json.dumps(alert))
```

`list.append` 在 CPython 是原子操作,跨线程也安全。

### 7.5 客户端宽容掉线

`recv_loop()` 用 `connection_alive: asyncio.Event` 标记连接状态。一旦掉线:
1. 主循环检测到 `connection_alive.is_set()` 为 False,退出
2. 摄像头自然释放
3. 用户重新启动客户端可自动重连 (下一版可加自动重连)

## 九、扩展思路

### 9.1 多客户端同时接入
当前架构天然支持:每个连接一个 session,session 间隔离。一台 4090 大概能并行处理 5-8 个客户端 (受 PoseConv3D 推理速度限制)。如果要更多,可以加 GPU 推理批量化。

### 9.2 二进制协议 (省带宽)
当前用 JSON。若网络极差可改 `struct.pack` + 二进制 frame:
- Header: `<IdII` (12 字节: frame_idx + timestamp + h + w)
- Per person: `<i4f51f` (208 字节)

5 人 × 30 FPS = 31 KB/s ≈ 250 kbps,比 JSON 还小一点。但目前 JSON 也够用,**不优化不必要的**。

### 9.3 报警声 / 推送
客户端 `recv_loop()` 收到 `alert` 时可以:
- `import playsound; playsound("alert.wav")`
- 调用钉钉/微信机器人 webhook
- 发短信

只需 `if mt == MSG_ALERT:` 分支加几行。

### 9.4 多摄像头
单客户端一个摄像头。多摄像头方案:多客户端 + 同一服务器,服务端按 `client_id` 区分,可在管理面板看每个摄像头实时状态。

### 9.5 录像与回放
客户端用 `--save-out recording.mp4` 保存带 overlay 视频。摔倒后可回看 (法律责任 / 医院告知用)。

## 十、常见问题排查

| 问题 | 可能原因 | 解决 |
|---|---|---|
| 客户端启动连不上 | 端口没暴露 | 检查 SSH 隧道或公网端口,`telnet host port` 测连通 |
| YOLO 启动慢 | 第一次下载权重 | 让 ultralytics 自动下载,或手动 wget |
| FPS 卡在 5 以下 | YOLO 模型太大 / imgsz 太大 | 改 yolo11n-pose + imgsz 416 |
| 摔倒不报警 | 服务端参数没全开 | 确认服务端日志显示 fall_trend / pose_heuristic / lost_track 都 on |
| 服务器 GPU 显存爆 | 多客户端同时连 | 限制最大并发,或换 4090/A100 |
| 网络延迟 > 500ms | autodl 在国外节点 | 换国内 autodl 节点 |
| 客户端窗口不显示 | 远程 SSH 跑客户端 | 客户端必须在能开窗口的物理机上跑;无窗口加 `--no-show --save-out video.mp4` |

## 十一、验证结果

用 elder_fall_7 真实数据(44 次推理 + 之后 YOLO 失败 162 帧)模拟客户端流式发送,服务端的 `FallDetectionSession.process_frame()` 在 **frame 131 触发报警**,原因 `fall_trend:slope_heur:slope=0.089,cur=0.36` — **与本地模式完全一致**。

这证明分布式部署保留了所有救 fall_7 的策略,且 fall_7 在网络架构下同样能 100% 救回。
