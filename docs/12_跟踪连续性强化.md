# 12 ‒ 跟踪连续性强化:解决 YOLO 跟丢导致的骨架断流

> 此前的所有改进 (FallTrendDetector / pose_heuristic / lost_track_alert) 都是**报警逻辑**的优化 — 在模型给出概率之后做兜底。但**真正的瓶颈**在于:YOLO Pose/Track 在顽固视频里容易跟丢,导致 PoseConv3D 拿不到连续骨架序列,模型根本没机会给出正确判断。本文档记录用 **骨架外推 + Kalman 运动模型** 直接解决"骨架断流"这个根本问题的方案。

---

## 一、问题诊断

### 1.1 跟丢的四种场景
| 场景 | 现象 | 持续时间 |
|---|---|---|
| 老人转身背对摄像头 | YOLO 漏检关键点,track 卡住 | 5-15 帧 |
| 椅子/家具遮挡 | YOLO 检测不到下半身,bbox 突然变小,score 暴跌 | 10-30 帧 |
| 快速摔倒过程 | 人体姿态剧烈变化,YOLO 来不及适应 | 5-10 帧 |
| 倒地后维持卧姿 | YOLO 失去人体特征,track 完全消失 | 至无穷 |

### 1.2 旧 pipeline 的缺陷
```
YOLO 失败的帧 → 这个 track 没出现在 track_ids 里
              → detector.update() 不会 push 这个 track 的 buffer
              → buffer 停滞在跟丢前的最后状态
              → 下次能推理时,buffer 里是"摔倒前的姿态"+"现在的时间戳"
              → PoseConv3D 看到的是"过时"的序列,无法捕捉摔倒瞬间
```

### 1.3 旧兜底为什么不够
原来的 `lost_track_alert` 是"track 死了之后基于历史 raw_prob/heur 判定"。这是**事后审计**,本质上承认了"模型已经没机会判断了"。FallTrendDetector 同理 — 也是基于已有的(可能不全的)概率序列。

**根本问题**:让模型在跟丢期间也能持续推理,而不是让兜底策略给模型"擦屁股"。

---

## 二、解决方案

三个新组件,组合解决问题:

### 2.1 SimpleKalmanBoxTracker — 运动状态记忆
每个 track 维护一个 Kalman 滤波器:
- **状态**: `[cx, cy, vx, vy]`(中心 + 速度)
- **更新**: 每次真实 detection 来了,用观测的 cx/cy 更新,EMA 平滑速度
- **预测**: 跟丢期间用 `predict_bbox(frame_idx)` 推算当前应在哪儿,带 `damping` 阻尼(预测越远越保守)
- **零外部依赖**: 不用 filterpy,~80 行自实现

### 2.2 PoseInterpolator — 骨架外推器
当 track 短暂跟丢时,从历史外推未来帧:
```python
extrapolate_one(history_kpts, history_scores, history_bboxes, gap, kalman_predicted_bbox)
```
- **关键约束**: 输出严格保持 COCO 17 点格式、原图坐标系
- **score 衰减**: 每外推 1 帧 score *= 0.85 → 模型自动降低对外推帧的信任,不会被假数据骗
- **Kalman 锚定**: 把外推骨架的整体平移锚定到 Kalman 预测的 bbox 中心(混合 0.6 Kalman + 0.4 线性外推)
- **自动放弃**: 超过 `max_extrapolation_frames=8` 就不再外推(认为 track 真的丢了,交给 lost_track_alert 接管)

### 2.3 增强 TrackMerger — 同人接力
旧版只用 `last_bbox` 做 IoU/中心距离匹配。强化为:
- **Kalman 预测匹配**: 同时尝试用 `tomb.kalman.predict_bbox(current_frame)` 做匹配
- **加权打分**: 静态 last_bbox 权重 1.0,Kalman 预测 0.9(略低避免过度)
- **`max_gap_frames` 15 → 60**: 容忍 2 秒内的复活

### 3.2 `inference/multitarget_realtime_demo.py` 改动

**TrackState 新增字段**:
```python
kalman: Optional[SimpleKalmanBoxTracker] = None
n_interpolated_frames: int = 0       # 已连续外推帧数
total_interpolated_frames: int = 0   # 整个生命周期累计 (诊断)
last_real_seen_frame: int = -1       # 最后真实 detection 帧 (区别于 last_seen_frame)
alert_source_tag: str = ""           # 'model'/'logic'/'trend'
```

**push 拆成两条路径**:
```python
def push(self, kpt, score, bbox, frame_idx):
    """真实 detection 喂入。"""
    self.buffer.push(kpt, score)
    self.bbox = bbox
    self.recent_bboxes.append(bbox.copy())
    self.last_seen_frame = frame_idx
    self.last_real_seen_frame = frame_idx
    self.n_interpolated_frames = 0   # 重置
    if self.kalman is None:
        self.kalman = SimpleKalmanBoxTracker(bbox, frame_idx)
    else:
        self.kalman.update(bbox, frame_idx)

def push_interpolated(self, interp_frame, frame_idx):
    """外推路径: 虚拟帧,不更新 last_real_seen_frame。"""
    self.buffer.push(interp_frame.kpts, interp_frame.scores)
    self.bbox = interp_frame.bbox
    self.recent_bboxes.append(self.bbox.copy())
    self.last_seen_frame = frame_idx    # 视为"看到了"(避免被 stale 清掉)
    self.frames_since_infer += 1
    self.n_interpolated_frames += 1
    self.total_interpolated_frames += 1
```

**update() 主循环新增 "step 1.5"**:
```python
# 1.5 跟踪连续性强化
if self.pose_interpolator is not None:
    for tid, st in self.tracks.items():
        if tid in seen_now:           # 有真实 detection,跳过
            continue
        if st.kalman is None:          # 没运动历史,跳过
            continue
        gap = frame_idx - st.last_real_seen_frame
        if gap < 1:
            continue
        # Kalman 预测当前 bbox
        pred_bbox = st.kalman.predict_bbox(frame_idx)
        # PoseInterpolator 外推骨架
        interp = self.pose_interpolator.extrapolate_one(
            list(st.buffer.kpts)[-6:],
            list(st.buffer.scores)[-6:],
            list(st.recent_bboxes)[-6:],
            gap=st.n_interpolated_frames + 1,
            kalman_predicted_bbox=pred_bbox,
        )
        if interp is None:
            continue   # 超过 max,放弃
        st.push_interpolated(interp, frame_idx)

# 2. 推理调度: 改为"被 push 过(真实或外推)就有资格"
for tid, st in self.tracks.items():
    if not st.is_ready:
        continue
    was_pushed = (tid in seen_now) or (st.last_seen_frame == frame_idx)
    if not was_pushed:
        continue
    # ...原推理逻辑
```

## 五、模型选项 (二次优化)

> ⚠️ 升级骨骼模型不是主要改进,核心仍是连续性强化。模型只在 YOLO 给出**错的**骨架时帮上忙,对**完全没给**的帧无效。

**关键约束**: 输出必须兼容当前训练:
- COCO 17 点顺序
- 原图坐标系 (x, y 像素)
- score 分布与训练数据相近

### 5.1 YOLO11 系列(推荐,与 yolo26x 完全兼容)
| 权重 | 大小 | 客户端 CPU | 服务端 GPU | 精度 vs yolo11s | 备注 |
|---|---|---|---|---|---|
| yolo11n-pose.pt | 6 MB | 25-35 fps | 200+ fps | baseline -3% | 极快 |
| yolo11s-pose.pt | 22 MB | 12-18 fps | 150+ fps | baseline | 当前默认 |
| yolo11m-pose.pt | 50 MB | 5-8 fps | 80+ fps | +3% | 平衡 |
| yolo11l-pose.pt | 100 MB | 不可用 | 60+ fps | +5% | 服务端 |
| **yolo11x-pose.pt** | **200 MB** | **不可用** | **40+ fps** | **+7%** | **推荐服务端** |

切换方式:`--pose-weights yolo11x-pose.pt` (Ultralytics 自动下载)。**坐标系/骨架格式/score 分布完全兼容**,trained PoseConv3D 不需要重训。

### 5.2 yolo26x-pose (现有默认)
保持兼容。性能比 yolo11x 略低一点,但已经训练好的 PoseConv3D 是基于 yolo26x 的骨架训练的,理论上保持一致最稳。

### 5.3 不推荐的换法
- **RTMPose / DWPose**: 精度高但是 top-down 流程,需要单独的 detector + ID 关联,改动太大
- **ViTPose**: 推理太慢
- **MediaPipe**: 33 点格式,与训练的 COCO 17 点不兼容

## 七、验证证据

### 7.1 跟踪连续性单元测试
模拟人物匀速向右移动,在帧 30 突然 YOLO 失败:

| Phase | 帧 | n_interp | buffer_len | Kalman cx | 推理次数 |
|---|---|---|---|---|---|
| 1: 正常 | 0-29 | 0 | 30 | 540 | 0 (buffer 未满) |
| 2: 跟丢 | 30-37 | 1→8 | 31→38 | 600→638 | **4** ⬅️ 关键 |
| 3: 超限 | 38-39 | 8 | 38 (停) | 638 | 0 |
| 4: 恢复 | 40-42 | 0 (清零) | 41 | 真实 | 2 |

**核心结论**: 跟丢期间模型仍推理 4 次,旧逻辑这 8 帧是 0 次。

### 7.2 真实数据 (fall_7) 对照实验
| 配置 | 报警帧 | reason |
|---|---|---|
| 旧逻辑 (无外推) | 131 | `fall_trend:slope_heur,src=trend` |
| 新逻辑 (有外推) | 131 | `fall_trend:slope_heur,src=trend` |
| 跟丢期间总外推 | — | **42 帧** |

fall_7 没"提前"是因为它已经被 fall_trend 救回了 — 报警来源是趋势策略,不是模型。PoseInterpolator 的价值在 **其他可能跟丢但摔倒尚未明显的视频** 上才会体现 — 模型本身能持续给出 raw_prob,不再 100% 依赖兜底。

### 7.3 行为不变性
| Invariant | 状态 |
|---|---|
| COCO 17 点顺序 | ✓ 严格保持 |
| 坐标系 (原图像素) | ✓ 严格保持 |
| score 分布 | ✓ 真实帧不变,外推帧自动衰减 |
| 不破坏旧报警 | ✓ 同帧同 reason |
| 兼容现有训练 | ✓ 不需要重训 PoseConv3D |

## 八、为什么这个方案是"对症下药"?

| 旧方案 | 思维 | 局限 |
|---|---|---|
| FallTrendDetector | 后处理:看模型已有结果做兜底 | 跟丢期间没有结果,兜底也没用 |
| lost_track_alert | 事后审计:track 死了之后用历史判定 | 太晚,可能错过摔倒瞬间 |
| pose_heuristic | 单帧判断:用单帧姿态判定 | 没有 buffer 喂时不工作 |
| **PoseInterpolator** | **前置修复: 让 buffer 不断流** | **直接解决根因** |

**对比示例: 假设老人 12 帧内完成摔倒,中间 6 帧 YOLO 失败**
- 旧逻辑: PoseConv3D 看到的是"摔倒前 6 帧 + 摔倒后跳跃 6 帧",时间断裂,可能输出 0.2
- 新逻辑: PoseConv3D 看到的是"摔倒前 6 帧 + 外推 6 帧 + 摔倒后 6 帧",时间连续,可能输出 0.6+(模型能识别下降趋势)
